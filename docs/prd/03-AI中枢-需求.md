# AI 中枢 · 产品需求

AI 中枢是系统大脑,负责 **理解 → 检索 → 生成 → 决策(回答 / 转人工 / 调用工具)**。

## 1. 能力地图

```
                    ┌──────────────────────────────┐
                    │          AI 中枢              │
                    │                              │
   入消息 ─►  预处理(脱敏/语种)  ─►                 │
                    │             │                │
                    │             ▼                │
                    │  ┌──────── 决策器 ────────┐   │
                    │  │ 路由: FAQ / RAG / 工具 / 兜底/ 转人工 │
                    │  └──────────┬───────────┘   │
                    │             ▼                │
                    │  ┌─────────────────────┐     │
                    │  │ FAQ 精确/相似匹配     │     │
                    │  └──────┬──────────────┘     │
                    │         ▼                    │
                    │  ┌─────────────────────┐     │
                    │  │ RAG: 向量检索 + 重排  │     │
                    │  └──────┬──────────────┘     │
                    │         ▼                    │
                    │  ┌─────────────────────┐     │
                    │  │ Prompt 编排 + LLM 生成 │   │
                    │  └──────┬──────────────┘     │
                    │         ▼                    │
                    │  ┌─────────────────────┐     │
                    │  │ 工具调用 (Function)  │     │
                    │  └──────┬──────────────┘     │
                    │         ▼                    │
                    │  后处理(敏感词/置信打分/引用) │
                    └─────────┬────────────────────┘
                              ▼
                          出消息 / 升级
```

## 2. 模块详述

### 2.1 预处理

- 脱敏: 手机号、身份证、银行卡号、邮箱使用占位符,在 LLM 上下文中以 `[PHONE_1]` 等替代,生成后回填(可配置)。
- 语种检测: 默认中文,支持英文(可扩展);非支持语种回退人工。
- 长度截断: 单轮用户输入 > 4000 字符提示精简或拆分。
- **直播/点播 上下文注入**: 将 `live_context`(直播间 / 节目 / 主播 / 清晰度 / CDN / 网络 / 卡顿事件 / 设备 / 用户角色)结构化拼到 System Prompt;上下文每次请求都重新注入(可能动态变化);保证敏感字段(具体 stream_url、IP)不落 LLM。

### 2.2 决策器 (Router)

按以下优先级判定动作:

1. 命中"必转人工"规则(关键词 / 黑名单 / VIP / 多轮重复未解决) → `HANDOFF`
2. 命中 FAQ 节点(用户点击常见问题叶子,或文本与 FAQ 题目精确/高相似匹配 ≥ 阈值) → `FAQ_ANSWER`(直接返回预设答案,不走 LLM)
3. 命中工具(订单查询、物流追踪、积分查询等触发词或槽位齐全) → `TOOL_CALL`
4. RAG 召回置信度 ≥ 阈值 → `RAG_ANSWER`
5. 通用闲聊 / 兜底 → `LLM_GENERAL`
6. 兜底失败 / LLM 自陈"无法回答" → `HANDOFF`

输出可解释字段(供后台调优):`{ action, reason, confidence, hits[] }`。

> 不再单独维护意图小模型与情绪模型。"该不该转人工"由 **规则 + 检索置信度 + 重复轮次 + 用户主动点按** 综合判断;"该回什么"由 FAQ / RAG / LLM 串联负责。如后续报表需要意图/情绪标签,改为离线批跑 LLM 标注(成本可控、不阻塞在线链路)。

### 2.3 FAQ 通道

- 数据来源: 管理后台「常见问题」配置的多级树(详见 C 端 §3.5)。
- 命中方式:
  - **精确匹配**: 用户输入与某叶子 `title` 完全一致(如来自 FAQ 卡片点击) → 直接命中。
  - **相似匹配**: 用户输入向量与所有叶子 `title` 余弦相似度 ≥ 0.86(可配置) → 命中。
  - **手动反馈**: 答案末尾"仍未解决"按钮命中失败时,将该问题计入"待标注"。
- 命中后直接返回叶子的 `answer.content_md` + 附件 + 延伸问题 + 动作按钮,延迟 < 100ms,**不消耗 LLM Token**。

### 2.4 RAG (检索增强生成)

- **知识库**: 多类目(QA 对、文档、政策、产品手册),支持按租户/技能组隔离。
- **入库流程**:
  1. 文档清洗 → 切片(语义切分,默认 512 tokens / 50 重叠)
  2. 向量化(BGE-M3 / OpenAI text-embedding-3-large 任选)
  3. 写入向量库 + 倒排(BM25)
- **检索**:
  - Hybrid: 向量 Top 50 + BM25 Top 50 → RRF 融合 → Reranker (BGE-Reranker) Top 5
  - 元数据过滤: 类目、有效期、生效区域
- **回答规范**:
  - LLM Prompt 强制"仅根据下面的资料回答,如无资料,回答`无法回答`"。
  - 答案末尾附引用 `[来源:产品手册-退货政策 §3]`,前端可点击展开原文。
  - `无法回答` 触发兜底(转人工或通用 LLM)。
- **缓存**: 高频问题(归一化文本 + 知识库版本)结果缓存 24h,命中直接返回(显著降本)。

### 2.5 Function / Tool 调用

- **工具注册表**: 后台白名单管理,每个工具有 schema、权限、超时、降级。
- **内置工具(直播/点播领域)**:
  - `get_play_diagnostics(uid, room_id|vod_id)` 拉取服务端 QoE(首帧、卡顿率、错误码、CDN 5xx 比例),并合并客户端上报的 `live_context.play`
  - `get_room_info(room_id)` 直播间快照(主播、状态、清晰度档位、是否限流、是否风控封禁)
  - `get_vod_info(vod_id)` 点播节目信息(版权方、上下架状态、字幕、是否会员专享)
  - `get_membership(uid)` 会员等级、到期、连续包月状态、家庭组成员
  - `get_subscription_orders(uid, days?)` 近期订阅/续费/打赏流水
  - `cancel_subscription(uid, sub_id)` 取消连续订阅(需用户在卡片二次确认)
  - `apply_refund(order_id, reason)` 申请退款(写操作,默认 dry_run + 二次确认;未成年退款必须人工)
  - `get_anchor_info(anchor_id)` 主播档案、签约状态、违规记录(供主播咨询)
  - `report_content(room_id|vod_id, type, evidence)` 内容举报立案
  - `switch_quality_hint(level)` 给客户端下发"建议切清晰度",由用户确认
  - `create_ticket(...)` 创建工单(异步处理类问题)
  - `get_user_profile()` 用户画像与标签
- **执行**:
  - LLM tool_choice → 后端校验参数 → 调用 → 结果再喂回 LLM 总结
  - 写操作类工具默认 `dry_run=true`,需用户在卡片上点"确认"。
- **超时**: 单次 ≤ 3s,超时降级为"系统繁忙,正在为你转人工"。

### 2.6 Prompt 编排

System Prompt 模板(简化示例):

```
你是「{brand}」官方智能客服(直播/点播平台),语气专业、礼貌、简洁,使用中文回答。

- 仅依据【知识资料】与【实时业务数据】回答,资料不足请直接说「我帮你转人工」。
- 不得承诺资料外的优惠/赔偿/时效;不解释平台审核细节、风控规则原文。
- **播放问题排查必须先调用 `get_play_diagnostics`**,根据卡顿/错误码 → 给具体建议(切清晰度、切网络、重进直播间);若是 CDN/源站/鉴权类错误码,直接转技术支持。
- 涉及订阅/退款/打赏/会员必须先调用 `get_membership` / `get_subscription_orders` 核实;**未成年人打赏退款一律转人工**。
- 涉及内容举报、版权、主播封禁申诉,直接转对应技能组。
- 回答 ≤ 200 字,卡顿建议 ≤ 3 步、分点。

【用户画像】{profile_json}
【直播/点播上下文】{live_context_json}      // 节目/主播/清晰度/CDN/网络/卡顿事件
【会话摘要】{summary}
【实时业务数据】{tool_results_json}
【知识资料】{rag_chunks}
```

模板由后台维护,支持版本化、A/B 实验,可按场景(`live_room` / `vod` / `anchor` / `settings`)分别配置。

### 2.7 后处理

- **敏感词**: 政治、暴恐、色情、竞品 → 直接替换 `***` 或拒答 + 转人工。
- **格式化**: 自动渲染列表、链接、表格(限定 Markdown 子集)。
- **置信打分**: 综合 RAG 相似度 + LLM logprob + 工具结果完整度,作为是否提示"转人工"的依据。
- **回填脱敏**: 还原占位符。

## 3. 转人工协议(交接给座席)

转人工时,中枢生成 **Handoff Packet** 推给座席:

```json
{
  "session_id": "...",
  "user": { "id": "...", "level": "VIP", "tags": ["新客","高复购"] },
  "summary": "用户咨询订单 #123 物流停滞 4 天,语气强烈、要求加急;曾在 03-21 投诉过同类问题(由 LLM 生成的纯文本摘要)",
  "reason": "rag_low_confidence | user_request | repeat_unresolved | rule_keyword | rule_blacklist | tool_failure",
  "entities": { "order_id": "123" },
  "history": [ { "role": "user/ai", "content": "...", "ts": ... } ],
  "suggested_replies": ["亲您好,我立刻为您加急联系...", "..."],
  "knowledge_hits": [ { "title": "...", "url": "..." } ]
}
```

## 4. 训练 / 调优闭环

- **采集**: 所有会话写入数仓(脱敏后),标记是否命中 FAQ、是否转人工、是否解决、用户评分。
- **离线标注**: 由 LLM 批跑给历史会话打"问题分类 / 用户态度"标签(替代在线意图/情绪模型),用于报表与 FAQ 优化。
- **人工标注**: AI 训练师在管理后台对低分/兜底会话标注正确答案,沉淀进 FAQ 或知识库。
- **回归**: 每次 FAQ / 知识库 / Prompt 变更,自动跑回归测试集(≥ 500 条),通过率 ≥ 95% 才能上线。
- **A/B**: 支持按用户分桶切流不同 Prompt / 模型,看在线指标。

## 5. 模型与供应

### 5.1 默认配置

- **对话 LLM 默认走 OpenAI ChatGPT API**(`https://api.openai.com/v1/chat/completions`),默认模型 `gpt-4o-mini`(性价比基线),复杂会话可在后台切到 `gpt-4o` / `gpt-4.1`。
- 协议层完全遵循 OpenAI Chat Completions(messages、tools、stream、tool_choice、response_format),其它供应商通过适配器实现同一协议。
- API Key 存放在管理后台「AI 配置 → 模型与路由」,后端读取后用 KMS 加密落库,**绝不下发到前端**;支持每个租户独立 Key。

### 5.2 模型管理(后台可配置)

后台配置项(每条记录即一个"模型档位",供路由引用):

```jsonc
{
  "id": "openai_default",
  "provider": "openai",            // openai / azure_openai / anthropic / deepseek / qwen / custom_openai_compatible
  "base_url": "https://api.openai.com/v1",
  "api_key_secret_ref": "kms://...",
  "model": "gpt-4o-mini",          // 自由填写,跟随 OpenAI 官方型号
  "params": {
    "temperature": 0.3,
    "top_p": 1.0,
    "max_tokens": 800,
    "presence_penalty": 0,
    "frequency_penalty": 0,
    "response_format": { "type": "text" }
  },
  "timeout_ms": 15000,
  "stream": true,
  "rate_limit": { "rpm": 600, "tpm": 200000 },
  "fallback_id": "openai_4o",      // 失败时切到的下一档
  "tags": ["default","cn-zone-a"]
}
```

- **多档**: 至少配置 `default` / `complex` / `cheap` 三档;路由按场景或 RAG 命中难度自动选择。
- **多供应商**: ChatGPT 官方、Azure OpenAI、自建 OpenAI 兼容(vLLM / Ollama / DeepSeek / 通义),只需 `provider + base_url + api_key`。
- **Anthropic 等非 OpenAI 协议**: 由 LLM Router 内部做协议转换,后台呈现仍是统一字段。
- **健康检查**: 每分钟 ping `/v1/models`(或最小 chat),失败计数累计触发自动降级到 `fallback_id` 并告警。
- **Key 校验**: 保存前调一次小请求验证可用,失败不允许保存。
- **试聊**: 后台模型档位详情页提供"试聊"窗口,直接命中该档位调试 Prompt。

### 5.3 推荐档位预设

| 档位 | provider / model | 用途 |
|------|------------------|------|
| default | openai / gpt-4o-mini | 在线对话默认,平衡成本与效果 |
| complex | openai / gpt-4o | 复杂推理 / 多工具调用 |
| cheap | openai / gpt-4o-mini 或 自建 OpenAI 兼容 | 高频简单场景、缓存预热 |
| embed | openai / text-embedding-3-large | RAG 向量化 |
| rerank | bge-reranker-v2-m3(自部署) | RAG 重排 |

### 5.4 兜底与切换

- 单档失败率 > 30% 自动切 fallback;全部失败则中枢返回 `HANDOFF` + Banner 告知。
- 限流: 租户级 RPM / TPM,会话级 QPS;超限给"系统繁忙"+ 转人工。
- 成本预算: 后台可设月度 Token 预算,达 80% 告警,达 100% 强制切 cheap 档或全人工。

## 6. 性能与降级

- 单次完整 AI 应答 P95 ≤ 2.5s(命中缓存 ≤ 200ms)。
- 流式输出: SSE/WS 流式推 token,首字节 ≤ 600ms。
- LLM 故障 → 切备用模型;全部失败 → 自动转人工 + Banner 告知。
- 限流: 单会话 QPS ≤ 1,租户级 QPS / Token 双维度配额。

## 7. 可观测性

- 每次推理写一条 trace: `{action, hits, provider, model, latency, tokens_in, tokens_out, cost, confidence}`。
- 指标: FAQ 命中率、AI 拦截率、首响、平均轮次、转人工率、用户满意度、Token 成本/会话。
- 报警: 拦截率周环比下降 > 5pt、单档失败率 > 5%、Token 预算消耗速率异常 — 自动告警。
