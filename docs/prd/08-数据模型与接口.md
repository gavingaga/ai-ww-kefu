# 数据模型与接口

## 1. 核心实体 (ER 简表)

```
Tenant ─┬─< Site ─< Channel
        ├─< User (C 端)
        ├─< Agent (坐席)
        ├─< SkillGroup ─< AgentMembership
        ├─< Announcement
        ├─< QuickReply
        ├─< FaqTree ─< FaqNode (self-ref, parent_id)
        ├─< KnowledgeBase ─< Document ─< Chunk
        ├─< PromptTemplate
        ├─< ModelProfile (LLM 档位 / API Key)
        ├─< Tool
        └─< Session ─< Message
                     ├─< AITrace
                     ├─< Handoff
                     └─< Survey
```

### 2.4 announcement

### 2.5 quick_reply
`id, tenant_id, scene, label, payload, icon, sort_order, target_segment(json), status`

### 2.6 faq_tree / faq_node (MySQL)

`faq_tree`: `id, tenant_id, scene, version, status, target_segment(json), updated_by, updated_at`

`faq_node`:
| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint PK | |
| tree_id | bigint | 所属树 |
| parent_id | bigint NULL | 顶层为 NULL |
| title | varchar(200) | 节点标题(用户可见) |
| icon | varchar(64) | emoji 或 icon key |
| sort_order | int | 同级排序 |
| is_leaf | tinyint | 1=叶子(QA) |
| answer_md | text | 叶子的答案 Markdown |
| attachments | json | `[{type,url,...}]` |
| follow_ups | json | 关联其它叶子 ID |
| actions | json | `[{type:"send_text|handoff|open_link|open_form", ...}]` |
| synonyms | json | 同义问列表(用于相似匹配) |
| status | enum | draft / online / offline |
| valid_from / valid_to | datetime | 生效时间 |
| created_at / updated_at | datetime | |

叶子节点的 `title` 与 `synonyms` 在保存时由后端调嵌入接口写向量(独立 collection,与知识库 chunk 隔离),用于在线相似匹配。

### 2.7 model_profile (MySQL,AI 模型档位)

| 字段 | 类型 | 说明 |

| model | varchar(128) | 自由文本(`gpt-4o-mini` 等) |
| params | json | temperature/top_p/max_tokens... |
| stream | tinyint | 是否流式 |
| timeout_ms | int | |
| rpm / tpm | int | 限速 |
| fallback_id | varchar(64) | 失败 fallback 档位 |
| tags | json | `["default","cheap","embed",...]` |

> 默认初始化记录:`openai_default`(provider=openai, model=gpt-4o-mini, tags=["default"]),首次部署后管理员只需在后台填入 API Key 即可启用。

### 2.10 ai_trace
`id, session_id, message_id, action(faq/rag/tool/llm/handoff), confidence, hits(json), tools(json), prompt_version, provider, model, latency_ms, tokens_in, tokens_out, cost`

### 2.11 handoff
`id, session_id, reason, packet(json), created_at, picked_at, agent_id`

### 2.12 knowledge_base / document / chunk
- `chunk`: `id, doc_id, kb_id, content, vector(向量库存)、metadata(json), version, status`

### 2.13 play_diagnostic_snapshot (MongoDB,会话维度)

每次调 `get_play_diagnostics` 与客户端上报合并的诊断快照,供坐席回放与离线分析:

`session_id, message_id, ts, client(json: 60s 日志), server(json: QoE)、verdict(本地网络/CDN/源站/鉴权/未知)、suggested_action(json), correlated_room_metrics(json)`。

## 4. WebSocket 协议

URL: `wss://ws.example.com/v1?token=...`

帧格式:
```json
{ "type": "msg|ack|ping|event|error", "seq": 12, "ack": 11, "ts": 1700000000000, "payload": { ... } }
```

### 4.1 客户端 → 服务端

- `msg.text`: `{ session_id, client_msg_id, content }`
- `msg.image`: `{ session_id, client_msg_id, url, width, height, size }`
- `msg.file`: `{ session_id, client_msg_id, url, name, size, mime }`
- `msg.read`: `{ session_id, up_to_seq }`
- `event.typing`: `{ session_id, typing: true|false }`
- `event.handoff`: `{ session_id }`
- `ping`

### 4.2 服务端 → 客户端

- `msg.*` (同上,role 区分 ai/agent/system)
- `msg.chunk` 流式 token: `{ session_id, msg_id, chunk, end:false }`
- `msg.recall`: `{ session_id, msg_id }`
- `event.queue_update`: `{ session_id, position, eta_seconds }`
- `event.agent_join`: `{ session_id, agent: {...} }`
- `event.agent_leave`: `{ session_id, reason }`
- `event.session_close`: `{ session_id, reason }`
- `event.announcement_update`: `{ items: [...] }`
- `event.quick_reply_update`: `{ items: [...] }`
- `event.faq_update`: `{ tree: {...}, version: 12 }`
- `event.faq_card`: `{ msg_id, tree_id, root_nodes: [...] }`(欢迎语后下发)
- `event.live_snapshot`: `{ msg_id, room_id|vod_id, title, anchor, qualities }`(直播间快照卡片)
- `event.play_diagnostic`: `{ msg_id, summary, verdict, suggested_action, raw? }`(播放诊断卡片)
- `event.bridge_call`: `{ name: "switchQuality"|"reenterRoom", args }`(由 AI/坐席请求宿主执行 JSBridge 动作,需用户在卡片上确认)
- `error`: `{ code, message }`
- `pong`

### 4.3 Ack / 重传

- 每条 server → client 消息带 seq;客户端定期回 `ack`。
- 网络恢复后客户端发 `pull { session_id, after_seq }` 拉缺漏。

## 6. AI 内部接口

- `POST /ai/infer` `{ session_id, history, profile, tools, prompt_version, profile_id?, stream }`
  - 返回 SSE / chunked,事件: `decision`(faq/rag/tool/llm/handoff)、`tool_call`、`tool_result`、`token`(流式 chunk)、`done`,以及 trace meta(provider/model/tokens/cost)。
- `POST /ai/embed` `{ texts: [...] }`
- `POST /kb/search` `{ kb_id, query, filters, top_k }` → `{ chunks: [...], scores: [...] }`
- `POST /faq/match` `{ tenant_id, query, scene }` → `{ hit: true|false, node_id?, score? }`(精确/相似匹配)

LLM Router 内部统一遵循 OpenAI Chat Completions 协议:

```
POST {base_url}/chat/completions
Headers: Authorization: Bearer {api_key}
Body:    { model, messages, tools?, tool_choice?, stream, temperature, ... }
```

非 OpenAI 协议(Anthropic 等)由 Router 适配为同一入参/出参。

## 7. 幂等与一致性

- 所有写接口必须支持 `Idempotency-Key`。
- 消息插入唯一约束 `(session_id, client_msg_id)`。
- 会话状态变更走状态机,非法跃迁返回 409。

## 8. 鉴权与签名

- C 端: JWT(短 TTL 30min)+ 刷新 token;支持匿名访客降级 token。
- 内部服务: mTLS + 服务账号 token。
- 上传签名: 短时(5min) STS,目录隔离 `tenant/{id}/...`,文件读取走 CDN + 签名 URL。
