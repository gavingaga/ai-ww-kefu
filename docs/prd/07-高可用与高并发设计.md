# 高可用与高并发设计

目标: **单集群百万级在线长连接、十万级并发会话、99.95% SLA、消息 P95 < 300ms。**

## 1. 容量目标 (V1)

| 维度 | 目标 |
|------|------|
| 在线长连接 | 500,000 并发 |
| 活跃会话 | 100,000 并发 |
| 消息 QPS(总) | 50,000 |
| AI 推理 QPS | 5,000(峰) |
| 全链路 P95 | ≤ 300ms(非 AI) / ≤ 2.5s(AI) |
| 数据持久化吞吐 | 写 30k/s,读 100k/s |
| **大主播开播突发** | 5min 内会话量 10x、平均首响仍 ≤ 30s |
| **赛事直播峰值** | 单赛事会话同比平峰 20x,人工排队等待 ≤ 90s |

## 2. 长连接网关设计

- **单节点目标**: 50k WS,内存 ≤ 4GB,P99 推送 < 50ms。
- **语言**: Go(goroutine + epoll)或 Java Netty。
- **会话路由**:
  - 客户端登录后从 Routing 服务获取一个 Gateway 节点(GSLB + 一致性哈希,按 user_id)。
  - Redis 维护 `session_id → gateway_node_id` 映射,用于跨节点定向推送。
- **心跳**: 客户端 25s ping;服务端 30s 无包断开;Idle close。
- **粘性**: 4 层(LB) 按客户端 IP+UA hash 落到同一节点;断连重连可换节点(Redis 重建映射)。
- **背压**: 单连接发送队列上限 1k 条,溢出丢弃低优先级或断开。
- **协议**: 自定义 JSON 帧,含 `type, seq, ack, payload`,客户端基于 seq 做去重 + 缺漏拉取。

## 3. 消息可靠性

- **At-least-once + 幂等**: 客户端发送带 client_msg_id;服务端入库唯一索引;重传幂等。
- **写入路径**:
  1. Gateway 收消息 → 校验 → 写 Redis Stream(`chat.in.{shard}`)
  2. Session 服务消费 → 写入 MongoDB(主) + Kafka(异步进数仓)
  3. Session 落库后 → 推送 Kafka `chat.out.{tenant}` → 目标 Gateway 推送
- **顺序**: 同一会话的消息写入用 `session_id` 哈希 → 分区,保证分区内顺序。
- **未读/离线**:
  - C 端断线期间消息进 `pending:{user_id}` Redis Stream + MongoDB。
  - 重连后客户端用最大 seq 拉取增量。

## 4. 横向扩展

| 层 | 扩展方式 | 关键点 |
|----|----------|--------|
| Gateway | 无状态(状态在 Redis),按连接数 HPA | 一致性 hash,扩缩容时少量迁移 |
| 业务服务 | K8s Deployment + HPA(CPU/QPS) | 服务无状态,熟客分配可走 Sticky Hint |
| MySQL | 读写分离,核心表分库分表(按 tenant + 时间) | 分片键提前规划 |
| MongoDB | Sharded Cluster,分片键 `{tenant_id, session_id}` | 冷热分层,7 日热,90 日温,>90 转 OSS |
| Kafka | 增加 partition;消费者组动态扩 | 消费幂等 |
| Redis | Cluster + 分槽;热点 key 加随机后缀 | 防止大 key |
| 向量库 | Milvus 分片副本 | 按租户隔离 collection |

## 5. 容灾与故障域

- **多 AZ**:同城 3 AZ,任意 1 AZ 宕机不影响服务(N+1 容量冗余)。
- **跨 Region**:异地热备(数据异步同步,RPO ≤ 5min,RTO ≤ 30min);灾难时通过 GSLB 切流。
- **Region 内蓝绿**:发布只对 50% 节点滚动;失败 30s 内自动回滚。
- **依赖隔离**:LLM、第三方工具、内容安全 API 全部走熔断器(Sentinel),失败率 > 30% 触发熔断 + 降级。

## 6. 限流与防刷

- **租户级**:总 QPS、并发会话、Token/天 配额。
- **用户级**:单用户每分钟消息数(默认 60)、文件上传(默认 30/h)。
- **IP/设备**:风控(连续断连重连、注册风暴、机器人特征)。
- **AI 级**:每会话 AI 轮次上限,达上限提示用户精简或转人工。
- **WAF**:封禁恶意 UA、SQLi/XSS、超大 payload。

## 7. 缓存策略

- **多级**: 本地(Caffeine)→ Redis → DB。
- **配置类(公告/快捷按钮/Prompt)**: 全量加载到本地缓存,变更通过 Redis Pub/Sub 推送失效。
- **热点 FAQ 缓存**: `hash(intent + normalized_query) → answer`,TTL 24h。
- **画像缓存**: 用户在会话期间内缓存 30min。

## 8. 数据库高可用

- **MySQL**:MGR 三副本 + ProxySQL,自动选主;每日全量 + 实时 binlog 备份到 OSS。
- **MongoDB**:Replica Set + Sharded Cluster;Oplog 异地同步。
- **Redis**:Sentinel + 多 AZ;持久化 RDB + AOF。
- **Kafka**:3 副本 + min.insync=2,跨 AZ 分布。

## 9. 关键服务降级

| 触发 | 降级 |
|------|------|
| LLM 不可用 | 自动转人工 + Banner 提示;AI 建议回复也禁用 |
| 知识库不可用 | LLM 走通用 + 警示文案,严重时直接转人工 |
| 工具系统超时 | 跳过工具,转人工或诚实告知"系统繁忙" |
| 数据库主写故障 | 切只读模式,新消息写 Redis Stream + Kafka,稍后回放 |
| Gateway 节点失联 | Routing 标记下线,客户端重连自动换节点 |
| 公告推送故障 | C 端按上次缓存展示 |

## 10. 蓝绿 / 金丝雀发布

- 后端: K8s Argo Rollouts,按流量比例 5% → 25% → 50% → 100%,每步监控错误率/延迟,自动回滚阈值。
- 前端: CDN 多版本共存,通过 cookie / header 命中灰度;支持指定用户 ID 灰度。
- 配置: 公告 / 快捷按钮 / Prompt 强制走灰度发布,有"立即上线"二次确认。

## 11. 直播/点播 突发流量预案

直播业务的流量曲线极不平稳: 大主播开播、签到红包、赛事直播都会在 1~5 分钟内推高到平峰 10~20 倍。

- **预热扩容**:
  - 与运营日历(开播表 / 赛事时间)联动,提前 30 分钟把 Gateway / Routing / AI-Hub HPA 最小副本数拉到 1.5x 预期。
  - LLM 档位提前预热,避免冷启动首字节飙高。
- **流量识别 + 削峰**:
  - 同一 `room_id` 进客服并发 > 阈值 → 触发"群发模式":自动给 C 端推荐"故障公告"卡片 + 禁用 AI 长生成,只走 FAQ + 工单留言。
  - 短时大量重复"卡顿"会话 → 自动触达运维 + 推抢修公告,客服侧用同一回复模板批量响应。
- **降级开关**(后台一键):
  - 关 AI 流式 → 改为缓存 / FAQ-only(降本 + 降时延)
  - 关 AI 建议回复(座席侧)
  - 关 RAG → 走人工
  - 启用"工单兜底": 排队 > 阈值时引导用户留下问题,异步回复
- **弹性资源**:
  - GPU/LLM Token: 与多供应商签弹性配额;`fallback` 路由自动切到便宜档位。
  - 出口带宽 / CDN: 与公司核心直播流量隔离,独立 CDN 配额。

## 12. 演练

- 月度 Chaos:随机杀 Gateway 节点、断 Redis 主、丢包模拟。
- 季度故障演练:Region 切换、LLM 全断、DB 主切。
- 性能压测:基于历史峰值 1.5x 目标,JMeter + 自研 WS 压测脚本。

## 13. 监控关键指标 (SLI)

- **可用性**: 5xx 比例、WS 连接成功率
- **延迟**: 消息端到端、AI 首字节、AI 完整、API P95/P99
- **吞吐**: QPS、并发连接、活跃会话
- **业务**: AI 拦截率、转人工率、首响、放弃率、满意度
- **直播/点播**: 同房间客服并发(异常飙升报警)、CDN 节点 5xx、播放诊断 API 健康分、退款/未成年退款件数、举报立案数
- **资源**: CPU/Mem/网络、GC、慢查询、Kafka lag
